# Explanations — Лабораторная работа №5 (RNN vs LSTM, IMDb)

## Раздел 1 — Концепции

### 1. Последовательности и почему обычные нейросети (Dense) не справляются

Текст — это **последовательность**: порядок слов определяет смысл. Фраза «фильм не плохой» и «плохой не фильм» содержат одни и те же слова, но означают разное.

Обычная полносвязная сеть (Dense/MLP) принимает на вход вектор фиксированной длины и обрабатывает все элементы **одновременно**, без учёта порядка. Если подать ей bag-of-words представление текста, она увидит набор слов, но не их последовательность. Из-за этого Dense-сеть не может уловить:
- зависимости между словами («не» + «плохой» = позитив);
- контекст, который накапливается по мере чтения предложения;
- структуру языка (подлежащее → сказуемое → дополнение).

Рекуррентные сети (RNN/LSTM) обрабатывают токены **по одному**, сохраняя «память» о предыдущих шагах в скрытом состоянии — именно это позволяет им учитывать порядок.

---

### 2. Токенизация: как текст превращается в числа

**Токенизация** — это процесс разбиения текста на элементарные единицы (токены) и присвоения каждому токену числового индекса.

В лабораторной используется `Tokenizer` из Keras:
1. `fit_on_texts(train_texts)` — анализирует обучающие тексты и строит **словарь** (word_index): каждому уникальному слову присваивается индекс (1, 2, 3, ...), отсортированный по частоте.
2. `texts_to_sequences(texts)` — заменяет каждое слово в тексте на его индекс из словаря.

**OOV (Out-Of-Vocabulary)** — специальный токен `<OOV>` с индексом 1. Он заменяет слова, которых нет в словаре (например, слова из тестовой выборки, не встречавшиеся в обучающей). Без OOV такие слова просто исчезли бы из последовательности, что исказило бы длину и структуру текста.

**Зачем ограничивать словарь (num_words=25000)?** Полный словарь IMDb содержит ~90 000 слов, но большинство из них встречаются 1–2 раза. Включение редких слов:
- раздувает Embedding-матрицу (больше параметров → больше памяти и времени);
- не даёт модели статистически значимой информации (одного примера мало для обучения эмбеддинга).

---

### 3. Паддинг: зачем выравнивать длины

Нейросеть обрабатывает данные **батчами** — группами примеров. Батч представляется как матрица, а матрица требует одинаковой длины строк. Отзывы IMDb имеют разную длину (от нескольких слов до тысяч), поэтому нужно привести их к единому размеру.

`pad_sequences(maxlen=256, padding='post', truncating='post')`:
- **maxlen=256** — все последовательности приводятся к длине 256 токенов. Это компромисс: достаточно для захвата основного смысла, но не слишком много для скорости обучения.
- **padding='post'** — если отзыв короче 256 токенов, нули добавляются **в конец**. Модель сначала видит реальные слова, а потом нули.
- **truncating='post'** — если отзыв длиннее 256 токенов, обрезается **конец**. Начало обычно содержит основную мысль автора.

Информация теряется при обрезке длинных отзывов, но это неизбежный компромисс. Нули при паддинге модель учится игнорировать (Embedding слой для индекса 0 возвращает нулевой вектор).

---

### 4. Эмбеддинги (Embedding)

**One-hot кодирование** слов при словаре в 25 000 даёт вектор длиной 25 000, где только один элемент равен 1, а остальные — 0. Это крайне неэффективно:
- Огромная размерность (25 000 вместо ~128);
- Все слова равноудалены друг от друга — нет понятия «похожести» слов;
- Разреженные вектора плохо обрабатываются нейросетями.

**Embedding** — это обучаемая матрица размером `(vocab_size × embedding_dim)`, то есть `(25000 × 128)`. Каждая строка — плотный вектор из 128 чисел, представляющий одно слово.

Размерности матрицы Embedding:
- **input_dim = 25000** — количество слов в словаре (строки матрицы);
- **output_dim = 128** — размерность эмбеддинга (столбцы матрицы);
- **input_length = 256** — длина входной последовательности.

В процессе обучения матрица Embedding подстраивается так, чтобы семантически близкие слова (например, «отличный» и «прекрасный») получали близкие векторы.

---

### 5. RNN: формула и принцип работы

Рекуррентная нейронная сеть обрабатывает последовательность **пошагово**. На каждом шаге t она получает:
- текущий входной вектор x_t (эмбеддинг текущего слова);
- скрытое состояние с предыдущего шага h_{t-1}.

**Формула скрытого состояния:**

```
h_t = tanh(W_h · h_{t-1} + W_x · x_t + b)
```

где:
- W_h — матрица весов для скрытого состояния;
- W_x — матрица весов для входа;
- b — вектор смещения;
- tanh — функция активации (сжимает значения в диапазон [-1, 1]).

**ASCII-схема развёртки RNN во времени:**

```
    x_1         x_2         x_3               x_T
     |           |           |                 |
     v           v           v                 v
  +------+   +------+   +------+           +------+
  |      |   |      |   |      |           |      |
  | RNN  |-->| RNN  |-->| RNN  |-->  ...  -->| RNN  |--> h_T (выход)
  |      |   |      |   |      |           |      |
  +------+   +------+   +------+           +------+
  h_0→  h_1→     h_2→      h_3→              h_T
```

Одна и та же ячейка RNN (с одинаковыми весами W_h, W_x, b) применяется на каждом шаге — это и есть «разделение весов во времени». Финальное скрытое состояние h_T (при return_sequences=False) подаётся в Dense-слой для классификации.

---

### 6. Затухающий градиент

При обратном распространении ошибки через время (BPTT — Backpropagation Through Time) градиент проходит через цепочку умножений:

```
∂L/∂h_1 = ∂L/∂h_T · ∂h_T/∂h_{T-1} · ... · ∂h_2/∂h_1
```

Каждый множитель ∂h_t/∂h_{t-1} включает производную tanh (максимум = 1) и матрицу W_h. Если собственные значения W_h < 1, то при умножении через много шагов градиент **экспоненциально затухает** → модель не может обучиться на далёких зависимостях.

Если собственные значения W_h > 1, градиент наоборот **взрывается** (exploding gradient) — но это решается gradient clipping. А вот затухание — фундаментальная проблема, для которой нужна другая архитектура (LSTM/GRU).

На практике SimpleRNN «забывает» информацию из первых ~10–20 токенов. Для отзывов длиной 256 токенов это серьёзная потеря.

---

### 7. LSTM: гейты и cell state

LSTM (Long Short-Term Memory) решает проблему затухающего градиента с помощью **cell state** (C_t) — отдельной линии памяти, через которую информация проходит с минимальными изменениями.

Три гейта управляют потоком информации:

**Forget gate (гейт забывания):**
```
f_t = σ(W_f · [h_{t-1}, x_t] + b_f)
```
Выдаёт значения от 0 до 1 для каждого элемента C_{t-1}. Близко к 0 — «забыть», близко к 1 — «сохранить».

**Input gate (входной гейт):**
```
i_t = σ(W_i · [h_{t-1}, x_t] + b_i)
C̃_t = tanh(W_C · [h_{t-1}, x_t] + b_C)
```
i_t решает, какие значения обновить; C̃_t — кандидат на новую информацию.

**Обновление cell state:**
```
C_t = f_t ⊙ C_{t-1} + i_t ⊙ C̃_t
```
Старая память, отфильтрованная forget gate + новая информация, отфильтрованная input gate.

**Output gate (выходной гейт):**
```
o_t = σ(W_o · [h_{t-1}, x_t] + b_o)
h_t = o_t ⊙ tanh(C_t)
```
Решает, какую часть cell state выдать как скрытое состояние.

**Cell state как «транспортёр памяти»:** градиент через C_t передаётся почти без потерь, потому что обновление C_t — это поэлементное сложение и умножение (а не матричное умножение через W_h, как в RNN). Это позволяет LSTM помнить информацию через сотни шагов.

---

### 8. Метрики классификации

**Accuracy (точность):**
```
Accuracy = (TP + TN) / (TP + TN + FP + FN)
```
Доля правильных предсказаний. Простая и интуитивная, но **врёт на несбалансированных данных**: если 95% примеров — класс 0, модель, которая всегда говорит «0», получит Accuracy = 95%.

**В нашей лабораторной IMDb сбалансирован** (50/50), поэтому Accuracy здесь — надёжный показатель.

**Precision (точность/прецизионность):**
```
Precision = TP / (TP + FP)
```
Из всех предсказанных положительных — какая доля действительно положительных. Важна, когда цена ложного срабатывания высока (например, спам-фильтр: лучше пропустить спам, чем отправить важное письмо в спам).

**Recall (полнота):**
```
Recall = TP / (TP + FN)
```
Из всех реально положительных — какую долю мы нашли. Важна, когда цена пропуска высока (например, диагностика рака: лучше ложная тревога, чем пропущенная болезнь).

**F1-score:**
```
F1 = 2 · (Precision · Recall) / (Precision + Recall)
```
Гармоническое среднее Precision и Recall. Удобен как единый показатель, когда нужно учесть оба аспекта. F1 штрафует за сильный перекос между P и R: если Precision=1.0, а Recall=0.01, то F1 ≈ 0.02, а не 0.5.

## Вопросы с защиты

**Вопрос 1:** Вы используете `padding='post'`. А что будет, если поставить `padding='pre'`? Изменится ли качество модели и почему?
**Ответ:** При `padding='pre'` нули добавляются **в начало** последовательности. Для LSTM это может быть даже лучше: последнее скрытое состояние h_T будет формироваться на основе реальных слов, а не нулей. При `padding='post'` RNN/LSTM обрабатывает текст, а потом «прогоняет» через себя серию нулевых векторов, что может частично «размыть» накопленную информацию (особенно у SimpleRNN). На практике разница обычно невелика, потому что Embedding(0) возвращает нулевой вектор, и LSTM-гейты учатся пропускать нули. Но теоретически `pre`-паддинг для RNN/LSTM предпочтительнее.

**Вопрос 2:** Почему в Embedding-слое `input_dim=25000`, а не `len(tokenizer.word_index) + 1`? Это одно и то же?
**Ответ:** Нет, это **не** одно и то же. `tokenizer.word_index` содержит **все** уникальные слова (может быть ~90 000), а `num_words=25000` ограничивает словарь для `texts_to_sequences`. Если поставить `input_dim=len(word_index)+1`, мы выделим память под эмбеддинги ~90 000 слов, хотя реально используем только 25 000. Это пустая трата памяти. Значение `input_dim` должно совпадать с `num_words` в Tokenizer, чтобы размер Embedding-матрицы соответствовал реально используемому словарю.

**Вопрос 3:** Вопрос-ловушка: у LSTM в 4 раза больше параметров, чем у SimpleRNN с тем же числом units. Может быть, LSTM побеждает просто потому, что у неё больше параметров, а не из-за гейтов?
**Ответ:** Это справедливое замечание. У SimpleRNN(64) количество параметров рекуррентного слоя: (64+128)×64 + 64 = 12 352. У LSTM(64): это число умножается на 4 (четыре набора весов: forget, input, cell candidate, output), итого ~49 408. Чтобы проверить гипотезу, можно взять SimpleRNN(128) с примерно таким же числом параметров и сравнить. Однако на практике даже при равном числе параметров LSTM выигрывает на длинных последовательностях — дело именно в архитектуре гейтов и cell state, а не просто в количестве весов.

**Вопрос 4:** Что будет, если убрать слой `Dense(64, relu)` между рекуррентным слоем и выходным `Dense(1, sigmoid)`?
**Ответ:** Модель всё равно будет работать, но может потерять в качестве. Промежуточный Dense-слой добавляет нелинейность и позволяет модели комбинировать признаки из скрытого состояния более сложным образом. Без него 64-мерный вектор h_T просто линейно проецируется в одно число (через Dense(1)), что ограничивает выразительную способность. На простых задачах разница может быть незначительной, но на сложных — ощутимой.

**Вопрос 5:** Вы обучаете Embedding «с нуля». А что такое предобученные эмбеддинги (Word2Vec, GloVe) и когда они лучше?
**Ответ:** Предобученные эмбеддинги — это векторные представления слов, обученные на огромных корпусах текста (миллиарды слов). Они уже «знают» семантику: «king» - «man» + «woman» ≈ «queen». Их лучше использовать, когда **мало обучающих данных** — модель получает хорошие эмбеддинги «бесплатно». В нашей лабораторной 25 000 обучающих отзывов — этого достаточно для обучения эмбеддингов с нуля, и предобученные могут не дать заметного улучшения. Но при 1 000 примеров GloVe дал бы значительный прирост.

**Вопрос 6:** Почему вы используете `binary_crossentropy`, а не `categorical_crossentropy`? А если заменить — что сломается?
**Ответ:** `binary_crossentropy` ожидает один выходной нейрон с sigmoid (выход — вероятность класса 1). `categorical_crossentropy` ожидает N нейронов с softmax (выход — распределение вероятностей по N классам). Если заменить loss на `categorical_crossentropy` без изменения архитектуры (оставить Dense(1, sigmoid)), обучение пойдёт неправильно: функция потерь будет интерпретировать единственный выход как вероятность одного из одного класса, градиенты будут некорректными. Для корректной замены нужно: Dense(2, softmax) + one-hot кодирование меток + `categorical_crossentropy`.

**Вопрос 7:** Что такое `return_sequences` и когда его нужно ставить в `True`?
**Ответ:** `return_sequences=False` (по умолчанию) — рекуррентный слой возвращает только **последнее** скрытое состояние h_T, форма выхода: `(batch_size, units)`. `return_sequences=True` — возвращает скрытые состояния **для каждого шага**: h_1, h_2, ..., h_T, форма: `(batch_size, timesteps, units)`. True нужен, когда: 1) после рекуррентного слоя идёт ещё один рекуррентный слой (stacked RNN/LSTM) — ему нужна последовательность на входе; 2) задача требует выхода на каждом шаге (например, NER — разметка каждого слова, или seq2seq-декодер).

**Вопрос 8:** Вопрос-ловушка: вы токенизируете и строите словарь только на train. А что если в тестовых данных встретится слово, которого нет в train? Модель не упадёт?
**Ответ:** Нет, не упадёт — именно для этого задан `oov_token="<OOV>"`. Все слова из test, которых нет в словаре (или которые вышли за лимит num_words=25000), будут заменены на индекс OOV-токена (= 1). Модель «видит» их как одинаковое неизвестное слово. Если бы OOV не был задан, такие слова просто исчезли бы из последовательности — длина текста сократилась бы, а структура исказилась.

**Вопрос 9:** А что будет, если обучить токенизатор и на train, и на test данных? Это же даст больше слов — будет лучше?
**Ответ:** Это **утечка данных** (data leakage). Токенизатор, обученный на test, «видит» распределение слов тестовой выборки. Хотя в данном случае (словарь — это просто список слов) утечка минимальна, это **методологически неправильно**. В реальных задачах обучающие и тестовые данные должны быть строго разделены. Словарь строится только на train — это имитирует продакшн, где мы не знаем заранее, какие слова встретятся.

**Вопрос 10:** Вы используете `maxlen=256`. А что если средняя длина отзывов — 400 слов? Не теряете ли вы важную информацию?
**Ответ:** Да, теряем. При `truncating='post'` обрезается конец отзыва. Если ключевая мысль в конце («...но в целом фильм ужасный»), модель её не увидит. Варианты решения: 1) увеличить maxlen (512, 1024) — но это увеличит время обучения и потребление памяти; 2) использовать `truncating='pre'` — обрезать начало, сохранив конец; 3) использовать двунаправленный LSTM (Bidirectional), который читает текст в обе стороны. В нашей лабе 256 — разумный компромисс для учебных целей.

**Вопрос 11:** Чем GRU отличается от LSTM? Когда лучше использовать GRU?
**Ответ:** GRU (Gated Recurrent Unit) — упрощённая версия LSTM. У неё **два гейта** вместо трёх: reset gate и update gate. В GRU нет отдельного cell state — скрытое состояние h_t одновременно является и «памятью», и выходом. GRU имеет меньше параметров (в 3 раза больше, чем SimpleRNN, vs 4 раза у LSTM) → быстрее обучается. На практике GRU и LSTM показывают сопоставимое качество, но GRU чуть быстрее. GRU лучше выбирать, когда: нужна скорость, датасет небольшой (меньше параметров = меньше переобучения), последовательности не слишком длинные.

**Вопрос 12:** Если увеличить `units` в LSTM с 64 до 512, станет ли модель лучше?
**Ответ:** Не обязательно. Увеличение units: 1) увеличивает ёмкость модели — она может запоминать более сложные паттерны; 2) но также увеличивает риск **переобучения** — модель начнёт запоминать обучающие примеры вместо обобщения; 3) значительно замедляет обучение (количество параметров растёт квадратично от units). Для IMDb с 25 000 примеров 64 units — хороший баланс. С 512 units модель, скорее всего, переобучится без дополнительной регуляризации (Dropout, L2).

**Вопрос 13:** Почему в RNN используется tanh, а не ReLU как в Dense-слоях?
**Ответ:** ReLU в рекуррентных слоях опасен из-за **взрывающихся активаций**: скрытое состояние h_t рекурсивно передаётся на следующий шаг. Если ReLU не ограничивает значения сверху, h_t может расти экспоненциально с каждым шагом. tanh ограничивает выход в [-1, 1], что стабилизирует вычисления. В Dense-слоях такой проблемы нет, потому что данные не передаются рекурсивно — каждый слой обрабатывает вход один раз.

**Вопрос 14:** А что будет, если поставить `epochs=50` вместо 5? Станет ли модель точнее?
**Ответ:** Скорее всего — нет. При 50 эпохах модель начнёт **переобучаться**: train loss продолжит падать, но val loss начнёт расти. Модель запомнит обучающие примеры, но потеряет способность обобщать на новые данные. Это видно на графиках: если val loss начинает расти, пора остановиться. Для предотвращения используют **Early Stopping** — автоматическую остановку обучения, когда val loss перестаёт улучшаться. 5 эпох в нашей лабе — это достаточно для сходимости на IMDb.

**Вопрос 15:** Вопрос-ловушка: вы используете `validation_split=0.1`. Но данные не перемешиваются перед разделением. Что если первые 10% обучающей выборки — все позитивные?
**Ответ:** Это реальная проблема. Keras при `validation_split` берёт **последние** N% данных (а не случайные). Если датасет отсортирован по классам (сначала все негативные, потом все позитивные), валидационная выборка будет состоять только из одного класса — метрики валидации будут бессмысленными. Решение: перемешать данные перед обучением (`np.random.shuffle`) или использовать `validation_data=(X_val, y_val)` с явным разделением через `train_test_split` из sklearn. В случае IMDb из HuggingFace данные уже перемешаны, но полагаться на это — плохая практика.
